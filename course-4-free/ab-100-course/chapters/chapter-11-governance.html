Six Responsible AI principles

Microsoft's published RAI principles — the lens through which every AI solution gets evaluated. Each principle maps to specific technical controls.

⚖
PRINCIPLE 01

Fairness

AI systems treat all groups equitably across demographics and use cases.

ControlsBias testing · fairness evaluation · disparate impact review
🛡
PRINCIPLE 02

Reliability & Safety

Systems function as intended; prevent harm; predictable in edge cases.

ControlsEdge case testing · fallback behavior · hallucination detection
🔒
PRINCIPLE 03

Privacy & Security

Protect personal and organizational data through strong controls and minimization.

ControlsManaged identity · RBAC · DLP · encryption · minimization
♿
PRINCIPLE 04

Inclusiveness

Empower people of all abilities and backgrounds; remove barriers.

ControlsAccessibility testing · multilingual · diverse user research
👁
PRINCIPLE 05

Transparency

Understandable with clear disclosures on AI involvement and limitations.

ControlsAI disclosures · model cards · limitation docs · escalation paths
⚓
PRINCIPLE 06

Accountability

Organizations retain responsibility for AI decisions — humans accountable, not the AI.

ControlsOwnership · governance boards · audit trails · sign-off

Four-layer defense in depth

Microsoft's published architecture for securing AI agents. Each layer catches what others miss. Defense in depth means no single failure compromises the system.

▸ The 4-layer defense in depth model
LAYER 01
Identity & Access

First line of defense. Without identity, nothing else holds.

  • Managed identities per agent
  • RBAC at narrowest scope
  • Separation of duties
  • Access review cadence
LAYER 02
Data Governance

What the agent can see — and what it can leak.

  • Data boundaries (internal/public)
  • DLP on connectors
  • Sensitivity labels propagated
  • Residency enforcement
LAYER 03
Observability & Cost

You can't govern what you can't see.

  • Centralized logging
  • Quality, safety, cost dashboards
  • Anomaly & spend alerts
  • Tagged resources for attribution
LAYER 04
Threat Protection

Active defense — for AI-specific failure modes.

  • Prompt manipulation detection
  • Input/output filtering
  • Pre-prod red teaming
  • Incident response readiness

Identity foundations

Two pillars — managed identities with RBAC, plus separation of duties. Every agent gets unique identity, least privilege, separated roles.

PILLAR 01

Managed identities + RBAC

Unique identity per agent Per environment (prod, pre-prod, dev) with ownership metadata.
Managed identities — no secrets Remove embedded credentials, simplify rotation.
RBAC at narrowest scope Resource > resource group > subscription. Tighter is better.
Acting on behalf of user Propagate user permissions. Don't elevate.
Acting as itself Service role with only necessary actions.
PILLAR 02

Separation of duties

Distinct roles for the four control points. Approvals required for production publishing.

Maker

Builds agent

Publisher

Promotes to Prod

Env Admin

Manages tiers

Security Admin

Owns guardrails

⚠ KEY RULE The same person must never be both Maker and Publisher — that defeats the whole pattern.

Four data governance controls

What the agent can see, use, keep, and disclose. Four controls cover the data lifecycle from ingestion through retention.

⛓
CONTROL 01
Data boundaries
  • Separate internal vs public workloads
  • Keep confidential out of public agents
  • Honor data residency for region
  • Knowledge, logs, memory all in scope
🏷
CONTROL 02
DLP & sensitivity
  • DLP restricts connectors & actions
  • Sensitivity labels on knowledge
  • Surface highest label in responses
  • Block sensitive outbound flow
🗑
CONTROL 03
Retention & minimization
  • Define windows for logs, memory
  • Automate purge & anonymization
  • Minimum data principle
  • Redact PII at preprocessing
👁
CONTROL 04
Transparency
  • Disclose AI involvement to users
  • Document data usage
  • Provide data deletion paths
  • Required by RAI & regulation

Five AI vulnerability categories

The threat model unique to AI. Traditional security covers some of these but most are AI-specific. Each one needs explicit mitigation.

VULN 01
💉
Prompt manipulation
  • "Ignore previous instructions"
  • Deceptive context
  • Multi-step coercion
  • Hidden instructions in files
Sensitive data leak · unintended actions
VULN 02
🎲
Model behavior
  • Hallucinations / fabrication
  • Toxicity / harmful output
  • Poor safety boundaries
  • Biased reasoning
Compliance risk · trust erosion
VULN 03
📤
Data exposure
  • Indirect exposure via prompts
  • Unprotected logs / memory
  • Excessive permissions
  • Malicious input files
PII leak · IP exfiltration
VULN 04
🔑
Identity / RBAC gaps
  • Elevated agent privileges
  • Connector escalation paths
  • Downstream tool over-access
  • Anomalous elevation attempts
Privilege escalation · lateral movement
VULN 05
⚙
Agent / workflow
  • Tool use without guardrails
  • Misinterpreted instructions
  • Poor auditing & rollback
  • Unsecured external calls
Wrong actions taken · no rollback

Five mitigation strategies + layered defense

Mitigations corresponding to the vulnerability categories. Each mitigation is necessary but not sufficient — the layered defense principle is what makes it work.

▸ The 5 mitigation strategies
1
Activity monitoring

Logs, anomaly detection, risk dashboards.

2
Evaluation & testing

Prompt resilience, safety, version validation.

3
Identity & RBAC

Least privilege, managed identities, approvals.

4
Input/output filtering

Block code execution, strip embedded prompts, safety filters.

5
Layered defense

Combine all the above. The architectural answer.

LAYERED DEFENSE PRINCIPLE

"A layered approach reduces single point failure risk: identity and access boundaries · secure data governance · model evaluation and testing · monitoring and anomaly detection · incident response readiness." — Each layer compensates for what the others miss.

Four compliance domains

The regulatory layer. Each domain is what auditors will examine. Together they form the compliance evidence package.

DOMAIN 01
Data residency
  • Where prompts process — by region
  • Where logs store — telemetry, conversations
  • Preview features — may differ from GA
  • Cross-region inference — disable by default
  • Connectors & plugins — verify they don't cross boundaries
DOMAIN 02
Purview controls (M365 Copilot)
  • Sensitivity labels — restrict cross-tenant transmission
  • DLP rules — block sensitive data in prompts or outputs
  • Auditing & policy insights — for Copilot solutions
  • Interaction logs — follow residency rules
DOMAIN 03
Grounding access & tuning
  • Least privilege by default — grounding access
  • Role-aligned partitions — separate by function
  • Stewardship assigned — clear ownership
  • Tuning approval workflows — verified lineage
  • Restricted promotion — tuned models gated
DOMAIN 04
Audit trails — models & data
  • Models: registration, tuning, promotion, rollback
  • Data: ingestion, schema, refresh, residency events
  • Capture metadata, not content — avoid exposure
  • Immutable logs — separation of duties enforced
  • Retention: 90d low-risk, 12-24mo regulated, ∞ incidents

Incident response playbook

Five steps from detection through recovery. Pre-rehearsed for high-impact agents.

The 5 steps
  • Detect — anomaly alerts, user reports, dashboards
  • Disable — quickly disable the agent
  • Preserve — logs & artifacts for forensic analysis
  • Notify — pre-defined communication templates
  • Recover — rollback, then post-mortem
⚡ TABLETOP DRILL Run a tabletop exercise quarterly for high-impact agents. First time you do incident response should NOT be in production.

Course capstone scenario

Apply Chapters 1–11 end-to-end. Pharmaceutical, multi-region, regulated.

★ Course Capstone · Pair Discussion

PharmaCorp's Enterprise AI Assistant Program

"PharmaCorp is a global pharmaceutical company. 75,000 employees across 40+ countries. Subject to FDA, EMA, HIPAA, GDPR — full regulatory complexity. They're launching an enterprise AI assistant program with: (1) a custom medical research model in Foundry for drug discovery, (2) Copilot Studio agents for clinical trial coordination and regulatory submissions, (3) D365 Customer Service AI for patient support, (4) deployment across US, EU, APAC, and LATAM regions. 5,000+ users across business, scientific, and clinical roles. CIO has asked you, the AI architect, to design the responsible AI, security, governance, risk, and compliance framework for the program. 12-month rollout. The Audit Committee will review your framework next month."
Discuss with your partner
  • Which RAI principles are most critical here? Why? What goes in the Audit Committee briefing?
  • Design the 4-layer defense in depth across the three solution components. What's different between the medical research model and the patient-support agent?
  • Threat model — what are the top 3 AI vulnerabilities for this program? How do you mitigate each?
  • Multi-region compliance — GDPR + HIPAA + FDA validation requirements. How do you architect for residency and audit?
  • Audit trail requirements — FDA expects validation evidence. What gets logged, how immutable, retained for how long?
  • Governance RACI — who's accountable for what at PharmaCorp? Where do you place the AI Architect?
  • Day 1 incident — what's your response plan if the medical research model produces a hallucinated drug interaction recommendation? Walk through the 5 steps.
Guide · capstone discussion points
Critical RAI principles for pharma
Reliability & Safety: Highest priority — clinical decisions can harm patients. Hallucinated drug interactions = direct patient harm. Privacy & Security: HIPAA + GDPR — patient data is the most sensitive class. PHI plus medical history. Transparency: FDA validation explicitly requires explainability. Clinicians and patients must know AI is involved. Accountability: Pharma legal liability — humans on the hook for decisions, not the AI. Required by FDA submissions. Audit Committee briefing focus: Lead with reliability/safety case + clear accountability chain. Show layered defense, not single controls.
4-layer defense across the three components
Medical research model (Foundry): Strictest — Layer 1 isolated managed identities, Layer 2 PHI never leaves region, Layer 3 every inference logged with model version, Layer 4 red team monthly + adversarial input testing. Clinical trial Copilot Studio agents: High — same Layer 1/2 baseline, Layer 3 conversation logging with PHI redaction, Layer 4 prompt injection filtering on uploaded protocols. D365 patient-support AI: Standard — Layer 1/2 standard, Layer 3 normal Copilot telemetry, Layer 4 standard content safety. Lower risk because it's directing to humans, not making clinical calls. Match rigor to risk: Don't apply medical-research-model rigor to the patient-support chatbot — that's gold-plating that delays the program.
Top 3 vulnerabilities + mitigations
1. Model behavior — hallucinations in clinical context: Mitigation = grounding-only responses (no free generation), confidence thresholds with human handoff below threshold, regular evaluation against medical golden datasets. 2. Data exposure — PHI leakage: Mitigation = DLP on all connectors, sensitivity labels propagated through agent responses, tenant-level Purview policies, regional data boundaries. 3. Prompt manipulation — clinical trial uploads with hidden instructions: Mitigation = input sanitization for uploaded files, structured prompt templates with allowlists, output filtering for unexpected actions. The threat model is uniquely pharma: Insider risk (researchers exfiltrating IP), regulatory (audit failure), patient safety (clinical errors). Mitigations must address all three.
Multi-region compliance architecture
Per-region deployment: 4 regions × 4 environment tiers = 16 environments minimum. Foundry workspaces per region. D365 tenants per region. EU GDPR: EU patient data never leaves EU — even in Dev. Right-to-deletion mechanisms. DPO sign-off on every release. US HIPAA: BAA agreements verified. PHI access logged per individual. Minimum necessary access. FDA validation: Model versions immutable post-validation. Validation evidence per region. CSV (computer system validation) documented. APAC + LATAM: Country-by-country review — each has unique regulations. Default in-region, deny cross-border. Cross-region exceptions: Documented approval workflow, time-limited, audit trail required.
Audit trail for FDA validation
What gets logged: Every model registration, tuning operation, promotion, rollback, evaluation result, data refresh, sensitivity label change, residency event, access attempt. Plus identity per action. Immutability: Append-only audit store. Separate identity from production write access. WORM (write once read many) for FDA-validated artifacts. Retention: FDA-regulated artifacts — minimum 7 years post-decommission. EU GDPR conflicts here — work with legal on retention vs. right-to-erasure conflict. Format: Structured (JSON) with correlation IDs across systems. Time-stamped with role-based attribution. Capture metadata, not content: Critical — log that PHI was accessed, not the PHI itself. Protects from audit trail becoming the leak vector.
Governance RACI for PharmaCorp
AI Architect: Accountable for RAI review, vulnerability mitigation, defense-in-depth design, model card content. Responsible for technical reviews. Single point of failure risk — needs strong R partners. Chief Medical Officer: Accountable for clinical safety review of medical research model. Veto authority on production release. DPO (Data Protection Officer): Accountable for GDPR compliance, residency, retention conflicts. Required EU role. Compliance Officer: Accountable for FDA submissions, HIPAA BAAs, audit reporting. Security Admin (CISO team): Accountable for security baseline, threat protection, incident response readiness. Product Owner per agent: Responsible for business outcomes and the user-facing decisions the agent enables. Audit Committee: Reviews quarterly. Accountable to the Board.
Day 1 incident — hallucinated drug interaction
Step 1 Detect: Either anomaly alert (output flagged) or clinician report. Whichever first. Step 2 Disable: AI Architect approves immediate disable. Within 5 minutes. Don't wait for the post-mortem. Step 3 Preserve: Lock the model version, conversation log, grounding state, model cards. Forensic snapshot. Step 4 Notify: CMO immediately (clinical safety). DPO and Compliance Officer (regulatory). Affected clinicians (clinical impact). FDA if required by regulation. Use pre-defined communication templates — don't draft under pressure. Step 5 Recover: Roll back to last validated model version. Post-mortem within 72 hours. Determine if grounding data, model behavior, or prompt was the failure mode. Update controls before re-enable. Validation evidence regenerated before re-deployment. The architect's call: Bias toward disable when in doubt. Patient safety over operational continuity. Always.
