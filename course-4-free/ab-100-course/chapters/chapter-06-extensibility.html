M365 agent design — three steps

Microsoft's three-step framework for designing Microsoft 365 Copilot agents. Apply in order. Don't skip ahead to data connections before you've nailed behavior.

1
Understand the Core Problem

Define task · business goal · decisions · systems involved.

Single high-value need
  • Lead preparation
  • Case triage
  • Policy Q&A
  • Workflow routing
2
Define Agent Behavior

Role · allowed/disallowed capabilities · guardrails · escalation rules.

Examples
  • "Service triage assistant" role
  • Tone & content boundaries
  • Compliance limits
  • When to escalate to humans
3
Connect Data & Tools

Microsoft Graph data, connectors, actions, custom APIs — least privilege.

Sources
  • SharePoint, OneDrive, Teams
  • Built-in connectors
  • Custom APIs (only when needed)
  • Zero-trust boundaries

Four collaboration patterns

When multiple M365 agents work together, they follow one of four patterns. Pick based on whether agents work in series, in parallel, iteratively, or under a coordinator.

⇄
Sequential Workflow

Linear pipeline — each agent depends on the previous one.

Agent A → Agent B → User Review
⇉
Parallel Evaluation

Multiple agents analyze data simultaneously, results combined.

Agent A ╱╲ → Combined
Agent B ╱ ╲ Result
↻
Feedback Loop Iteration

Agent refines output iteratively until a quality threshold is met.

Draft → Review → Refine → ✓ Quality met
⊛
Orchestrated Interaction

Primary agent delegates subtasks based on intent or conditions.

Coordinator → { A | B | C } based on intent

Copilot Studio — four extensibility layers

Layer up only as far as you need. Most agents stay at instruction or skill level. Pro-code is the last resort, not the first.

LAYER 01 · CHEAPEST
✎

Instruction

Behavior, tone, boundaries, role through prompt modification.

"You are a finance triage agent. Always cite policy IDs."
LAYER 02
⊞

Skill

Modular operations agents can invoke. Reusable across agents.

"Lookup-Customer-Tier" skill reused in 5 agents
LAYER 03
⇆

Integration

Enterprise systems of record. Data retrieval, actions, external APIs, events.

Custom connector to legacy ERP for invoice approval
LAYER 04 · LAST RESORT
{ }

Pro-Code

VS Code authoring. Custom logic, reusable components, source-controlled.

Complex orchestration logic with deep transformations

Model Context Protocol — what & why

MCP standardizes how agents retrieve and interpret business context. Most relevant for Dynamics 365 Finance & Operations — but applicable wherever agents need authoritative business semantics.

Why MCP matters

The standardized context contract

MCP serves as a structured contract for how agents access and interpret business context. Exposes business entities, relationships, labels, metadata, and domain-specific objects — especially in Dynamics 365 F&O.

Reduces hallucination

Grounds agents in real F&O context.

Multi-app interoperability

Consistent semantics across agents.

Improves explainability

Decisions trace to authoritative data.

PATTERN A
Context-Driven Reasoning

Real-time MCP context for compliance-sensitive workflows. Finance, procurement, vendor management.

PATTERN B
Workflow-Integrated

MCP drives approvals, escalations, status summaries inside active workflows.

PATTERN C
Multi-Agent via MCP

Cross-domain collaboration — HR + Finance + Supply Chain agents share standardized context.

Computer Use — when to reach for it

Agents can control mouse, keyboard, and read on-screen UI elements. Use only when no API or connector exists — UI changes can break automation. Treat it as the API of last resort.

⊘
No API exists

The classic case. The system has no programmable interface.

  • Legacy systems
  • Vendor portals (no API)
  • Desktop-only apps
⌨
Repetitive UI tasks

High-volume tasks that follow the same UI sequence every time.

  • Data entry
  • Form submission
  • Copy-paste workflows
⇄
Multi-app navigation

Bridge systems that don't have native integrations.

  • Web → CRM data flow
  • Portal report download
  • Spreadsheet updates from UI
⚛
Human-like reasoning

Where the UI is dynamic and the agent must interpret context.

  • Dynamic page elements
  • Context-dependent menus
  • Vision + reasoning combined

Reasoning depth & voice mode

Standard reasoning is the default — fast and cheap. Deep reasoning unlocks multi-step problem solving. Voice mode adds the hands-free interaction layer.

▪ DEFAULT · FAST · CHEAP

Standard Reasoning

Conversational responses, summaries, simple calculations, basic assistance.

Use for
  • Conversational responses
  • Summaries & classifications
  • Simple workflows
  • ~90% of agent interactions
▪▪ ADVANCED · STRUCTURED

Deep Reasoning

Multi-step problem solving, business rule evaluation, structured decision trees.

Use for
  • High-precision logic
  • Multi-stage evaluation
  • Constraint validation
  • Decomposition tasks
Example: Multi-step financial validation, recommendation generation.
▪ VOICE / IVR

Voice Mode

Hands-free interaction via IVR. Different design rules from text agents.

Design rules
  • Short, clear phrasing
  • Confirmation steps
  • Handle interruptions
  • Confidence checks
Use cases: customer service, field tech, healthcare triage, IT self-service.

Scenario discussion

Choose the right extensibility approach. Justify with the layered framework.

Scenario · Pair Discussion

ContractFlow law firm wants a contract drafting agent

"ContractFlow has 200 attorneys. They draft commercial contracts daily. Standard clauses sit in SharePoint. The firm has a 50-page style guide for legal language. They use a 12-year-old paralegal database (no API) to look up case precedents. Senior attorneys want voice dictation while driving. Drafts must always cite the precedent. Compliance audits every clause. Project deadline: 14 weeks."
Discuss with your partner
  • Custom Foundry model, M365 Copilot agent, or extended Copilot Studio? Justify with the indicators.
  • Which extensibility layers (instruction · skill · integration · pro-code) would you use, and for what?
  • Where does Computer Use fit? Where does MCP fit? Where does Voice Mode fit?
  • What's the right reasoning depth — Standard or Deep — for clause drafting? Why?
Guide · suggested points
Right approach — Extended Copilot Studio (NOT custom Foundry)
Don't go custom Foundry — only one indicator hits cleanly (domain specificity). Insufficient justification for the cost and timeline. Extend Copilot Studio — instruction-level customization for legal style guide, skill-level for clause retrieval, integration for paralegal database via Computer Use. 14-week timeline rules out Foundry custom anyway.
Extensibility layers in use
Layer 1 (Instruction): Encode the 50-page style guide as agent instructions and tone constraints. Layer 2 (Skill): Modular skills — "Draft-NDA-Clause", "Find-Precedent", "Validate-Citation" — reusable across attorneys. Layer 3 (Integration): SharePoint connector for clause library; Computer Use for the legacy paralegal database. Layer 4 (Pro-code): Skip. Low-code is sufficient for this scope.
Computer Use · MCP · Voice Mode placement
Computer Use: Only for the 12-year-old paralegal database (no API) — log in, search precedents, extract data, return to agent. MCP: Lower priority — would add value if integrating with D365 F&O for billing, but not core to drafting itself. Voice Mode: Yes — senior attorneys want hands-free dictation. Design for short phrasing, confirmation steps, interruption handling.
Reasoning depth — Deep, but selectively
Standard reasoning for routine clause retrieval and Q&A — most interactions. Deep reasoning for clause-level decisions involving multi-jurisdiction analysis, conflict checking, or precedent reconciliation. Compliance-sensitive moments. Don't enable Deep reasoning by default — cost and latency hit isn't justified for simple retrieval.

Chapter 6 — key takeaways

Six things to carry forward into Chapter 7, where we cover orchestrating prebuilt agents.

1
Custom Foundry — 5 indicators: Domain · High-Impact · Sovereignty · Unique Workflows · Cost at Scale. Need 2+ to justify.
2
M365 agents — 3-step design: Understand Problem · Define Behavior · Connect Data & Tools. Strict order.
3
Copilot Studio — 4 extensibility layers: Instruction · Skill · Integration · Pro-Code. Use lowest sufficient.
4
MCP = standardized contract for business context. 3 patterns: context-driven · workflow-integrated · multi-agent.
5
Computer Use — API of last resort. Use only when no API exists or for UI-bound legacy systems.
6
Behavioral capabilities — Standard default · Deep for multi-step logic · Voice for hands-free interaction.
