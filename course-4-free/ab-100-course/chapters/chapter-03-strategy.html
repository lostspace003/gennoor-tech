CAF + Agent Lifecycle — unified

Two complementary frameworks. CAF gives you the platform foundation. The Agent Lifecycle gives you the operating model. Use them together to prevent agent sprawl and accelerate value.

Cloud Adoption Framework

End-to-end backbone for AI on Azure

The standard Microsoft methodology covering AI adoption from business outcomes to operations.

  • Six phases — Strategy, Plan, Ready, Govern, Secure, Manage
  • Foundation work — landing zones, networking, governance
  • Reference architectures and Well-Architected guidance
  • Applies to all AI workloads, not just agents
Agent Lifecycle

Operating model for AI agents

Layers on top of CAF to give teams a clear way to plan, govern, build, and operate agents at scale.

  • Plan agents — strategy and feasibility per use case
  • Govern & Secure agents — policies, roles, controls
  • Build agents — standard process, templates, CI/CD
  • Operate agents — telemetry, SLOs, lifecycle

Three agent type categories

A ladder. Start at the top — fastest, simplest, lowest risk. Climb down only when functional requirements force you.

Tier 1 · Start here

SaaS Prebuilt Agents

Immediate value, minimal customization
ExamplesM365 Copilot · Sales Copilot · Service Copilot · Finance Copilot · Researcher · Facilitator
Use whenProductivity scenarios. Standard workflows. Customer needs immediate value.
Tier 2 · Most projects

Low-Code (Copilot Studio)

Moderate customization, rapid deployment
ExamplesCustom topics · Knowledge sources · Custom connectors · MCP servers · Generative answers
Use whenSaaS doesn't quite fit. Need custom knowledge or topics. Business-led customization.
Tier 3 · Last resort

Pro-Code (Foundry / Containers)

Maximum flexibility, deep orchestration
ExamplesFoundry agents · Custom orchestration · Fine-tuned models · Containerized agents · Activity Protocol
Use whenComplex multi-agent flows. Custom logic. Strict isolation. Industry-specific agent.

The SaaS-first decision rule

Microsoft's official guidance. One question, two branches, one rule of thumb.

Does a SaaS agent meet the functional requirements?
Yes →
Use the SaaS agent

Configure, ground in customer data, ship. Most engagements stop here.

Outcome: fastest time to value, lowest risk, lowest cost.
No →
Choose Copilot Studio or Foundry

Default to Copilot Studio. Move to Foundry only when complexity demands it.

Decision criteria below — apply rigorously.
Decision criteria when SaaS doesn't fit: evaluate cost, effort, customization depth, data access requirements, and governance readiness. Move down the ladder only when one of these forces it. Document your reasoning so the next architect understands the choice.

UNIT THREE

Single vs Multi-Agent
Architecture

Most architects design too many agents too soon. Start with one. Scale only when evidence forces it.
14 / 25

When to use multi-agent — and when not to

Single-agent first. Always. Add complexity only when there's evidence the single agent can't handle the job.

Single Agent (start here)

Default architecture for bounded scenarios
  • Consolidates logic in one place
  • Reduces coordination overhead
  • Simplifies governance and monitoring
  • Faster to ship, easier to debug
Use whenSingle domain, single team, single security boundary. The job stays bounded.

Multi-Agent (when justified)

Specialized agents with separation of concerns
  • Decomposes responsibilities by domain
  • Enforces separation of concerns
  • Scales across teams and security domains
  • Each agent gets its own scoped permissions
Use whenCrossing security boundaries, multiple teams own data, roadmap demands expansion.
Choose multi-agent first only if: crossing security/compliance boundaries · multiple teams own distinct data · roadmap demands expansion · multiple dependencies exist. If none apply, stay single-agent and validate before adding orchestration.

Five orchestration patterns

From the Microsoft Agent Framework SDK. Treat orchestration as a workflow with state, branching, and error handling — not a brittle prompt-to-prompt daisy chain.

→
Sequential

Deterministic pipeline. Plan → enrich → verify → act.

Use for: staged tasks with clear handoffs
⫶⫶
Concurrent

Parallel agents on independent subtasks. Aggregate and reconcile.

Use for: research, evidence gathering
○○○
Group Chat

Agents propose; moderator arbitrates. Mediated conversation.

Use for: complex decisions, deliberation
⇥
Handoff

Transfer context to a specialist agent or human on escalation.

Use for: human-in-the-loop, escalations
✦
Magentic

Dynamic specialization. A 'magnet' pulls in expert agents at runtime.

Use for: open-ended, varied scenarios

Scenario discussion

Multi-agent design exercise — apply CAF, agent platform decisions, orchestration, and constraints.

Scenario · Pair Discussion

Automate RFP intake and response at NorthStar Consulting

"NorthStar receives 50 RFPs a week. Each takes a junior consultant 8 hours to extract requirements, draft initial response, and route to a senior reviewer. Some RFPs have confidentiality requirements. Some need pulling in compliance attestations. Different teams own different sections of every RFP response. Leadership wants AI assistance — and they want it 'soon.'"
Discuss with your partner
  • Single agent or multi-agent? Justify using the four conditions.
  • What roles would you assign? (Planner, Extractor, Drafter, Reviewer, etc.)
  • Map each role to a Microsoft platform — M365 Copilot, Copilot Studio, or Foundry. Why each?
  • Which orchestration pattern(s) would you use, and where?
Guide · suggested points
Single or multi-agent?
Multi-agent justified: crosses confidentiality boundary (some RFPs), multiple teams own sections, multiple data sources required. Three of four conditions met. But — start with the smallest viable multi-agent. Don't over-design.
Suggested roles
Planner — reads RFP, identifies sections needed, assigns to specialists. Extractor — pulls requirements, deadlines, eval criteria, compliance flags from the RFP doc. Drafter — generates first-draft response sections (technical, commercial, compliance). Reviewer — flags issues, missing info, escalates to human senior.
Platform mapping
Reviewer → Microsoft 365 Copilot — reviewer reads in Word/Outlook, comments in flow. Extractor → Copilot Studio — low-code, structured topics, integrates with internal RFP DB. Drafter → Copilot Studio — knowledge-grounded responses from past proposals (SharePoint). Planner / Coordinator → Foundry — pro-code orchestration, calls other agents, tool integration.
Orchestration pattern
Sequential for the main flow: Planner → Extractor → Drafter → Reviewer. Concurrent for evidence collection (multiple Drafters working different sections in parallel). Handoff to a human senior consultant when Reviewer flags issues — gate high-risk submissions. Constraints: confidential RFPs use scoped Foundry deployment with private networking; never cross to public model endpoints.
