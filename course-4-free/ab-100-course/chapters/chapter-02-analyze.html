⚡

Data Analytics

  • Natural language → insights

🎯

Decision-Making

Agents that recommend next moves
  • Scenario recommendations
  • Risk identification via patterns
  • Synthesize across docs, meetings, data
  • Human still owns the decision

UNIT TWO

Grounding — the five dimensions
of AI-ready data

If you only remember one thing from this chapter, remember this: weak grounding is the number-one cause of AI project failure.
09 / 20

What is grounding?

The mechanism that connects an AI agent to your trusted data instead of relying solely on the model's training data.

Definition

Grounding ties AI responses to your organization's real data

The agent doesn't make answers up from training data — it retrieves verified content from your SharePoint, OneDrive, Dataverse, or connected sources before generating a response.

Why it matters: reduces hallucinations, increases trust, ensures compliance, respects user permissions.

How Microsoft does it

The grounding pipeline

1. Semantic IndexBuilds semantic representations of all M365 content — Graph-wide.
2. Retrieval APIFetches the right content — scoped to user permissions.
3. GenerationModel uses retrieved content to write a grounded response.

The five dimensions of grounding data

Memorize these. You'll evaluate every customer's data against all five before designing anything.

DIMENSION 1
✓

Accuracy

Data reflects real, verifiable facts. SMEs have validated it. Authoritative sources back it up.

Test"Is this verified by an SME?"
DIMENSION 2
⊙

Relevance

Data matches the agent's intended use case, workflow, and business domain.

Test"Does this fit the use case?"
DIMENSION 3
⏱

Timeliness

Data is current. Reflects latest documents, policies, and conversations.

Test"When was this last updated?"
DIMENSION 4
✦

Cleanliness

No duplicates, clean structure, consistent formatting, minimal noise.

Test"Is the structure consistent?"
DIMENSION 5
⤓

Availability

Data is accessible, indexed in Microsoft Graph, with clear access controls.

Test"Can the right user reach it?"

Best practices for reviewing grounding data

These are your action items when a customer hands you a SharePoint with five thousand documents and says "ground the agent in this."

①

Evaluate content quality before grounding

Remove outdated or conflicting information before upload. Audit before ingest, not after deployment.

Example: Customer's SharePoint has 3 versions of the same expense policy. Pick the authoritative one, archive the rest.
②

Use SharePoint or OneDrive for authoritative content

Microsoft 365 semantic index covers Graph-connected sources. Network drives don't ground.

Example: Move HR policy library from a Network share into a curated SharePoint site labeled "HR-Authoritative."
③

Maintain consistent formatting and structure

Headings, metadata, predictable layout improve embeddings and retrieval precision.

Example: Apply a "Policy Document" template — same heading hierarchy, same metadata schema across all docs.
④

Review permissions and SME validation regularly

Permissions drift. Knowledge changes ownership. Quarterly review keeps grounding trustworthy.

Example: Quarterly audit — every grounded source has a named SME owner and reviewed access controls.

Microsoft databases — AI-ready

Modern Microsoft databases ship with vector search, embeddings support, and AI-native features. Choose based on your workload.

Vector data, semantic search, JSON storage. AI features inside the relational database your team already knows.

Best forExisting SQL Server shops adding RAG to apps.

Unified analytics platform with OneLake, Lakehouse, and AI workloads on one governed data plane.

Best forAnalytics + AI on the same governed estate.

The RAG pipeline — five stages

RAG is what separates a prototype from a production-trustworthy system. Every architect should be able to draw this from memory.

⤓
Ingest

Pull from SharePoint, OneDrive, databases, APIs

◇
Chunk & Embed

Split into passages, convert to vectors

▦
Index

Store vectors in Azure AI Search or Graph Index

🔍
Retrieve

Search index for relevant passages, scoped to permissions

✎
Generate

Model writes grounded response with citations

Why architects love RAG: real-time data access, preserved privacy, mitigated hallucinations. Swap the model anytime without rewriting the data layer.

Scenario discussion

Apply the five dimensions and the data architecture lens.

Scenario · Pair Discussion

BrightCorp wants an HR policy assistant

"We have 8,000 employees across 14 countries. HR policy questions overwhelm our HR team — same five questions, asked a thousand times a week. We have policies in SharePoint, but they're scattered across regional sites. Some are PDFs, some are Word, some are out of date. Leadership wants an agent rolled out broadly and fast."
Discuss with your partner
  • What types of data must be collected and categorized first?
  • Run the five-dimension test — which dimensions are likely weakest here?
  • Which Microsoft data architecture components do you need? (Database? Index? RAG?)
  • What is the single biggest risk if data is not governed before launch?
Guide · suggested points
What data must be collected and categorized first?
Authoritative HR policy library — the master version of every policy, version-controlled. Regional variations tagged with country/region metadata so the agent surfaces the right one. SME owner mapping — every policy has a named owner who validates accuracy. Sensitivity labels applied via Microsoft Purview — public, internal, confidential.
Which dimensions are weakest? (Five-dimension test)
Cleanliness — mixed formats (PDF, Word), scattered, duplicates likely. Major cleanup needed. Timeliness — "some are out of date." Stale policies = compliance risk. Relevance — regional content needs scoping so a German employee doesn't get US policy. Accuracy & Availability — likely OK once the cleanup is done; SharePoint Graph index handles availability.
Which architecture components do you need?
Centralized SharePoint site — single authoritative HR-Policy site, regional subsites. Semantic Index (Microsoft Graph) — automatic for SharePoint content; ground via Copilot Retrieval API. Copilot Studio agent with knowledge sources scoped to the HR-Policy SharePoint site + regional metadata filters. Microsoft Purview for sensitivity labels, lineage, and access policies. No custom RAG required — the M365 Copilot grounding pipeline handles this scenario natively.
Biggest risk if data is not governed?
Compliance / legal exposure — agent confidently quoting outdated policy is a liability. Trust collapse — one wrong answer to a sensitive question (parental leave, harassment policy) and adoption dies. Permission leakage — without proper access controls, agent could surface confidential HR docs to the wrong audience. Hidden fix is upstream — push back on leadership: 30 days of data governance saves 6 months of post-launch firefighting.
